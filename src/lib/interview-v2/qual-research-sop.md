## 이 문서의 목적

심층 인터뷰(IDI) + 스크리너 설문으로 구성된 소비자 정성조사를, **동일한 품질과 구조로 재현**하기 위한 규칙서다. 새 프로젝트를 시작할 때 이 문서를 그대로 적용하면 된다.

**적용 대상**: 소비재/헬스케어 카테고리의 IDI 기반 소비자 조사 (n=20~80, 2개 이상 세그먼트 비교)

# 0. 필요한 입력 파일

| 파일 | 형식 | 필수 | 비고 |
|---|---|---|---|
| **트랜스크립트** (세그먼트별) | `.txt` | 필수 | `PARTICIPANT N of M: {이름}` 구분자 필요 |
| **스크리너 응답** (세그먼트별) | `.csv` | 필수 | 인구통계 + 선별 문항 |

- 문항: `{번호}-{질문문구}` 형식. **복수응답은 보기별 개별 컬럼으로 분리되어 있음**

# 1. 분석 기준(base) 결정 규칙 ★가장 먼저 확인할 것

**⚠️ 이번 프로젝트 최대 실수: 스크리너 응답자 228명 중 인터뷰 완료 61명만으로 인구통계를 집계해 결론이 왜곡됐다.**

### 1-1. 기준을 4단계로 분리해서 파악한다

| 기준 | 정의 | 용도 |
|---|---|---|
| **전체 제출** | 스크리너에 응답한 전원 | 규모 파악용. **분석에 쓰지 말 것** |
| **함정 통과** | 부정응답 필터를 통과한 사람 | 중간 확인용 |
| **적격 (qualified)** | 선별 조건까지 통과한 사람 | **스크리너·인구통계 분석의 기준** |
| **인터뷰 완료** | 실제 인터뷰한 사람 | **인터뷰 내용(발화·객관식) 분석의 기준** |

### 1-2. 어떤 분석에 어떤 기준을 쓸 것인가

| 분석 대상 | 기준 |
|---|---|
| 스크리너 문항 (Q1~Qn) | **적격자 전원** |
| 인구통계 프로필 | **적격자 전원** |
| 인터뷰 내 객관식 | **인터뷰 완료자** |
| 인터뷰 발화·인용 | **인터뷰 완료자** |

### 1-3. 반드시 수행할 편향 검증

**인터뷰 표본이 적격자 전체를 대표하는지 반드시 교차 검증한다.**

```
인터뷰 표본 vs 적격자 전체를 나란히 놓고 비교
→ 행동·태도 문항: 대개 일치 (결론 유효)
→ 인구통계: 자주 어긋남 (연령·인종·소득·학력)
```

**어긋나는 항목이 있으면 보고서에 "정정 사항" 섹션을 명시적으로 만든다.** 조용히 고치지 말 것.

### 1-4. 함정 문항(trap) 탐지는 필수

스크리너에 **존재하지 않는 가짜 보기**가 심어져 있는지 확인한다.
- 가짜 브랜드명 (예: `Dermacalm RX Daily Shield`)
- 가짜 속성 (예: `Triple-Barrier genome match`)

이걸 고른 응답자는 **문항을 읽지 않은 것**이므로 전수 확인 후 제외 여부를 판단한다. (통상 7% 내외 발생. 이미 `qualified=False` 처리되어 있는 경우가 많음)

# 2. 데이터 처리 파이프라인

## 2-1. 응답자 분리
`PARTICIPANT N of M` 구분자로 트랜스크립트를 분할한다.

## 2-2. 프로필 매칭 (트랜스크립트 ↔ 스크리너)

**⚠️ 스크리너 CSV의 `participantEmail`은 대개 비어 있고, `lastName`은 이니셜(예: `W.`)만 있다.**

**매칭 키: `firstName`(전방일치) + `lastName` 이니셜**

```python
def match_row(full_name, df):
    fl = full_name.lower()
    tokens = full_name.split()
    last_initial = tokens[-1][0].lower()
    # 1차: firstName이 전체이름의 접두사 + 성 이니셜 일치
    cands = [i for i in df.index
             if fl.startswith(str(df.at[i,'firstName']).strip().lower())
             and str(df.at[i,'lastName']).strip().lower()[:1] == last_initial]
    if len(cands) == 1: return cands[0]
    # 2차: 첫 토큰 정확일치 + 성 이니셜
    ...
```

**엣지 케이스**: `firstName`이 두 단어인 경우(예: "Molly Jane") → 전방일치로 해결됨.

**매칭률은 반드시 100%를 목표로 하고, 결과를 보고한다.** (예: "61/61 매칭 완료")

## 2-3. 개별 파일 생성
응답자 1명 = 1파일. 구조:
```
# {이름}
## 프로필 (Profile)          ← 인구통계 표 + 스크리너 응답 표
## 인터뷰 응답 (Interview)    ← 트랜스크립트 전문 (원문 보존)
```
+ 세그먼트별 통합 파일 + zip 번들

## 2-4. 객관식(closed-ended) 문항 식별

**트랜스크립트 안에도 객관식 문항이 섞여 있다.** 판별 기준:
- 답변의 평균 길이가 짧음 (< 70자)
- 서로 다른 값의 개수가 적음 (보기 개수만큼)

**⚠️ 세그먼트별로 보기 문구가 다르게 설계된 문항이 있다.** (예: "다른 스킨케어처럼 매일" vs "아이의 루틴으로 매일")
→ 이런 문항은 **합산 시 왜곡되므로 반드시 표에 `*` 표시하고 주석을 단다.**

## 2-5. 주관식 코딩 ★가장 중요한 단계

**객관식만 집계하면 핵심을 놓친다.** 이번 프로젝트에서 가장 중요한 발견 3개가 전부 주관식에서 나왔다.

**반드시 코딩해야 할 주관식 문항 유형:**

| 유형 | 질문 예시 | 산출물 |
|---|---|---|
| **현재 사용 제품 스펙** | "지금 쓰는 제품을 자세히 말해주세요 — 브랜드·용량·가격·용기·제형" | 브랜드 침투율, 제형·용기·가격 분포, 병용 패턴 |
| **고통·삶의 영향** | "얼마나 심한가요? 잠·일·외출·자기 감정에 어떤 영향을?" | 고통 유형별 분포 |
| **첫 구매 계기** | "이 제품이 필요하다고 처음 느낀 순간은?" | 진입 트리거 분포 |
| **재구매 계기** | "가장 최근에 산 이유는?" | 재구매 동인 |
| **전환·이탈** | "브랜드를 바꾼 적 있나요? 계기는?" | 락인/전환 구조 |

**코딩 방법**: 61명 전원의 답변을 읽고 MECE한 카테고리로 분류 → 빈도 집계 → 대표 인용문 확보

# 3. 분석 프레임워크 (MECE 4축)

**모든 객관식 문항을 이 4개 축 중 하나에 배치한다. 축에 안 들어가는 문항이 있으면 축을 수정하되, 문항을 버리지 않는다.**

## ★ 0부는 반드시 '응답자 프로필'로 시작한다

**보고서 본문 맨 처음(1부 앞)에 인구통계 프로필 섹션을 넣는다.** 독자가 "누구 얘기인지" 모른 채 숫자를 읽으면 안 된다.

**0부에 반드시 포함할 것:**
- 그룹 정의 (누가 왜 다른 그룹인지 한 줄로)
- **한눈에 보기 표** (평균 나이 · 성별 · 인종 · 학력 · 고용 · 지역)
- 나이 / 성별 / 인종 / 학력·직업·소득 / 거주 지역 각각의 분포표
- 세그먼트 고유 변수 (예: 자녀 나이)
- **각 항목마다 "그래서 뭘 뜻하는지" 한 줄 해석을 붙인다** (예: "여성 85% + 단독 결정 96% → 구매 결정은 한 명의 여성이 혼자 내린다")
- **인터뷰 표본이 적격자 전체와 어긋나는 항목은 ⚠️ 로 표시하고 실제 수치를 병기한다**

## ★ 섹션 제목은 축 이름이 아니라 '질문'으로 단다

**"축 A", "축 B" 같은 내부 구조 명칭을 본문 제목에 노출하지 않는다.** 앞의 안내표를 안 읽고 본문부터 읽는 독자는 그게 뭔지 전혀 모른다.

| ❌ 이렇게 쓰지 말 것 | ✅ 이렇게 쓸 것 |
|---|---|
| 축 A. 질환 구조 | **1부. 이 병은 어떤 병인가 — 가끔 오는 일인가, 늘 있는 상태인가** |
| 축 B. 사용 맥락 | **4부. 어떻게 바르는가 — 내 피부인가, 가만있지 않는 아이 몸인가** |
| 축 C. 신뢰 구조 | **6부. 왜 그 제품을 사는가 — 효과 때문인가, 안전 때문인가** |
| 축 D. 접근 경로 | **7부. 제품을 어떻게 알게 되는가 — 추천인가, 검색인가** |

**제목만 읽어도 그 장(章)이 무엇을 다루는지 알 수 있어야 한다.** 상호 참조도 "→ 축 D" 가 아니라 "→ 7부" 로 쓴다.

```
[축 A] 질환/문제 구조 — 얼마나 자주, 얼마나 오래 겪는가   → 시장의 성격
   └ A 이어서 ①: 어떤 고통이 필요를 만드는가 (주관식 코딩)
   └ A 이어서 ②: 언제 사러 나가는가 — 진입/재구매 트리거 (주관식 코딩)
[축 B] 사용 맥락 — 누구의 몸에, 어떤 조건에서 쓰는가      → 제형·패키지 요구
   └ B 이어서: 지금 실제로 쓰고 있는 것 (제품 감사, 주관식 코딩)
[축 C] 신뢰 구조 — 무엇을 근거로 구매를 정당화하는가       → 메시지·인증 전략
[축 D] 접근 경로 — 어떤 문을 통해 제품에 도달하는가        → 채널·예산 배분
```

### 각 축의 서술 순서 (고정)
```
숫자(표) → 그 숫자의 의미 → 응답자의 목소리(인용) → 그래서 뭘 해야 하나
```

### 마지막에 반드시 넣을 것
1. **네 축 종합 대비표** (그룹별)
2. **두 그룹이 똑같이 원하는 단 하나** — 차이를 다 걷어냈을 때 남는 공통 니즈
3. **만만치 않은 걸림돌** — "지금도 불만 없다"는 응답자 비율
4. **실행 우선순위 표** (근거 숫자 + 인용 병기)
5. **부록: 더 확인해볼 것들** (이번 조사로 답 못한 질문)

# 4. 표 작성 규칙

## 4-1. 모든 정량 표는 3개 열을 갖는다
```
| 항목 | 전체 (n=총) | 그룹A (n=) | 그룹B (n=) | 차이(%p) |
```

## 4-2. 전체 수치의 함정을 반드시 경고한다
**두 그룹이 정반대인 항목에서는 전체 평균이 진실을 가린다.**

> 예: '신제품 고려 요인' 전체 1위는 '습진 전용'(28%)이지만,
> 실제로는 본인 1위 '즉효'(30%), 부모 1위 '습진 전용'(35%)로 정반대.
> → **평균이 두 개의 얼굴을 뭉갠 것.**

**→ 보고서 앞부분에 "표를 읽는 법" 섹션을 넣고, 차이(%p)가 큰 항목은 전체를 무시하고 그룹별로 보라고 명시한다.**

## 4-3. 해석 임계값
- **5%p 내외** → 노이즈로 간주. 해석하지 않는다.
- **15%p 이상** → 해석적 무게를 둔다.
- **0% 또는 100%** → 반드시 주목한다. (예: "확실히 바꾸겠다" 양육자 **0%**)

## 4-4. 한쪽 그룹에만 주어진 보기는 `*` 표시
전체 n으로 나누면 왜곡된다. 해당 그룹 기준 수치를 병기한다.

## 4-5. 복수응답은 합계 100% 초과를 명시

# 5. 인용문 규칙

## 5-1. 인용은 '장식'이 아니라 '증거'다
**숫자 1개당 인용 1~2개를 붙인다.** 숫자만 있는 섹션은 미완성이다.

## 5-2. 인용이 새로운 발견을 만든다
숫자로는 안 보이던 것이 인용에서 드러난다. 실제 사례:
- `manage(관리하다)` vs `treat(치료하다)` — 동사 하나가 세그먼트 차이의 본질
- *"scoop it up"* / *"dig out"* — 부모가 왜 통(jar)을 선호하는지 설명
- *"고장 안 났는데 왜 고쳐요?"* — "절대 안 바꿈 19%"의 정체

## 5-3. 형식 ★한국어 + 원어(local language) 반드시 병기

**인용문은 예외 없이 두 줄로 쓴다. 한국어만 쓰거나 원문만 쓰지 않는다.**

```
> "한국어 번역" — {이름}, {그룹}({추가정보})
> *"Original quote in the language the interview was conducted in"*
```

**왜 둘 다 필요한가**
- **한국어만** → 클라이언트가 원문을 검증할 수 없다. 번역자의 해석이 개입한 것처럼 보인다.
- **원어만** → 리서치 배경이 없는 독자가 못 읽는다.
- **둘 다** → 가독성과 검증 가능성을 동시에 확보한다.

**추가 규칙**
- 원어 인용은 **1~2문장**으로 자른다 (길면 `…` 처리)
- 자동 전사 원문이므로 **명백한 오류만 최소 보정**하고, 그 사실을 각주에 밝힌다
- ⚠️ **원문 추출 시 객관식 보기 텍스트가 섞이지 않도록 주의한다.** 트랜스크립트에서 객관식 답변은 구두점 없이 이어져 있어서, 문장 분리 시 `"Itching Diagnosed by a doctor Almost always present…"` 같은 쓰레기가 딸려온다. **보기 문자열 목록을 만들어 먼저 제거한 뒤 문장을 자른다.**

## 5-4. 브랜드명 전사 오류에 주의
음성 자동 전사는 브랜드명을 심하게 왜곡한다. **별칭 사전을 만들어 정규화한다.**

> 실제 사례: Eucerin → `Usurin / Zucerin / uterine / Eusteren / Ursarin / Usaren / urethane`
> Aquaphor → `Aqua for / Aquifer / Aqua Fur`
> Aveeno → `Avino / Innoveno / Baby Alino`

# 6. 문체·표현 규칙 ★클라이언트 전달용

**"전문적인 보고서 형식은 유지하되, 리서치 배경이 없는 사람도 읽고 이해할 수 있게."**

## 6-2. 과장된 비유 금지 ★가장 자주 걸리는 부분

**분석가는 발견이 흥미로울수록 비유를 쓰고 싶어진다. 참아야 한다.**
비유는 글을 멋있게 만들지만, **읽는 사람에게 새로운 정보를 주지 않는다.** 숫자와 사실만으로 이미 충분히 강하다.

### 금지 표현 → 대체 표현 (실제 교정 사례)

| ❌ 쓰지 말 것 | ✅ 이렇게 쓸 것 |
|---|---|
| $10~$20이 이 시장의 **한복판이다** / **심장이다** | 대부분 **$10~$20 사이에서 산다** |
| A 브랜드의 **독주** / **혼자 멀리 앞서 있다** | A를 쓰는 사람이 **압도적으로 많다** |
| **역설** / **앞뒤가 안 맞는 대목** | **싫다면서 계속 쓰고 있는 것이 있다** |
| **균열** / **틈** | **새로 드러난 부분** |
| 유일한 **무기** / **결정타** | 유일한 **강점** / (삭제) |
| **잊으면 안 되는 벽** / **만만치 않은 걸림돌** | **쉽지 않은 이유** |
| 브랜드 교체는 **도박이다** | **아이한테 위험을 무릅쓰는 일이다** |
| 30%p **갭 = 즉시 채울 수 있는 빈자리** | **30%포인트나 비어 있다. 지금 바로 채울 수 있는 자리다** |
| 평균이 두 얼굴을 **뭉개버렸다** | **정반대인 두 그룹을 평균 내니 둘 다 안 보이게 됐다** |
| **정면 공격**한다 / **비집고 들어간다** | **정면으로 맞붙는다** / **파고든다** |
| 잠깐 **열리는 문** / **탐색의 창** | **기회는 처음 한 번뿐이고, 놓치면 다시 오지 않는다** |
| **여기서부터 모든 게 갈린다** | **이 차이가 뒤에 나올 모든 이야기의 출발점이다** |
| **가장 뜻밖의 발견** | **예상과 가장 달랐던 것** |
| **여기서 반전이 나온다** | **그런데 예상 밖의 결과가 나온다** |
| **오프라인에서 승부가 난다** | **구매의 82%가 오프라인 매대에서 일어난다** |
| 소비자가 레지멘을 **조립한다** | 사람들이 **알아서 자기만의 조합을 짠다** |
| **지문처럼 남아 있다** / **~의 함수다** | (전부 삭제. 그냥 사실을 쓴다) |

### 자가 점검 질문
> **"이 표현을 빼고 숫자와 사실만 남겨도 뜻이 통하는가?"**
> 통한다면 → 비유를 뺀다. 통하지 않는다면 → 애초에 근거가 부족한 주장이다.

### 제출 전 필수: 금칙어 전수 검색
```
한복판 심장 독주 역설 균열 무기 결정타 빈자리 도박 승부 조립
뭉개 정면공격 지문처럼 ~의함수다 탐색의창 벽 틈
```
→ 검색 결과 **0건**이 될 때까지 고친다.

## 6-3. 섹션 제목은 '질문' 또는 '주장'으로
- ❌ "축 A. 질환 구조"
- ✅ "축 A. 습진이 **가끔 오는 일**인가, **늘 있는 상태**인가"
- ❌ "C-4. 굳어짐의 증거"
- ✅ "C-4. 부모는 한번 정하면 안 바꾼다 — 그리고 **0%**라는 숫자"

## 6-4. 각 섹션은 "그래서 뭘 해야 하나"로 닫는다

# 7. 실수 방지 체크리스트 ★이번에 실제로 틀렸던 것들

| # | 실패 사례 | 방지 규칙 |
|---|---|---|
| **1** | **압축 손실**: 정량·정성을 융합하면서 4축에 안 맞는 정량 내용(피부이슈 세부·진단율·도포부위·용량)을 통째로 잘라냄 | **융합 ≠ 삭제.** 객관식 문항 전부가 어느 축엔가 들어갔는지 체크리스트로 확인 |
| **2** | **기준 혼동**: 스크리너 228명인데 61명으로 집계 | **작업 시작 전에 base를 4단계로 확정**하고 문서 최상단에 명시 |
| **3** | **인구통계 편향 미검증**: 61명의 인종·연령이 전체와 달랐음 | **인터뷰 표본 vs 적격자 전체를 반드시 교차 검증** |
| **4** | **정규식 오염**: 문항 텍스트(보기 목록)까지 매칭돼 제형·용기가 100%로 나옴 | **응답(Answer/Follow-up A) 텍스트만** 대상으로 코딩. 문항 텍스트 제외 |
| **5** | **주관식 누락**: 트리거·고통·제품스펙 문항이 주관식이라 첫 집계에서 빠짐 | **문항 목록을 전수 확인**하고 주관식도 코딩 대상에 포함 |
| **6** | **일괄 치환 부작용**: "카테고리→시장" 치환으로 "병 관리 시장다" 같은 비문 발생 | 치환 후 **반드시 문법 검증 패스** |
| **7** | **보기 비대칭 무시**: 세그먼트별 보기 문구가 달라 0% vs 48%라는 가짜 극단값 | 보기 문구를 세그먼트 간 대조하고, 다르면 `*` 표시 + 합산 행 별도 제공 |
| **8** | **인용문 원어 누락**: 한국어 번역만 넣어서 검증 불가능한 보고서가 됨 | **한국어 + 원어 반드시 병기** (5-3 참조) |
| **9** | **원문 추출 오염**: 객관식 보기가 문장에 섞여 들어감 | 보기 문자열을 먼저 제거한 뒤 문장 분리 |
| **10** | **축 이름을 본문 제목에 노출**: 안내표를 안 읽은 독자는 "축 A"가 뭔지 모름 | **제목은 질문형으로.** 상호 참조도 "1부/4부"로 |
| **11** | **응답자 프로필 누락**: 누구 얘기인지 모른 채 숫자부터 읽게 됨 | **0부를 인구통계 프로필로 고정** |
| **12** | **과장된 비유 남발**: "시장의 한복판", "브랜드의 독주", "연고의 역설" 등 — 클라이언트가 반복 지적 | **6-2 금칙어 목록으로 전수 검색.** 발견이 흥미로울수록 비유 욕구가 커지니 의식적으로 억제 |

# 9. 작업 순서 요약

```
1. 파일 확인 → 트랜스크립트 구조·CSV 컬럼 파악
2. ★ base 4단계 확정 (제출/함정통과/적격/인터뷰완료)
3. 함정 문항 탐지
4. 응답자 분리 + 프로필 매칭 (100% 목표)
5. 개별 파일 + 통합 파일 + zip 생성
6. 객관식 식별·집계 (세그먼트별 + 전체 + Δ)
7. ★ 주관식 코딩 (제품스펙 / 고통 / 트리거 / 전환)
8. ★ 인터뷰 표본 vs 적격자 편향 검증
9. 인용문 추출 (숫자당 1~2개, 한국어 번역 병기)
10. 보고서 작성
    - 0부 = 응답자 프로필 (필수)
    - 1~7부 = 질문형 제목 (축 이름 노출 금지)
    - 각 절: 숫자 → 의미 → 목소리(한국어+원어 병기) → 액션
11. 문체 점검
    - 자카곤 금지어 대조 (6-1)
    - **과장된 비유 금칙어 전수 검색 → 0건 확인 (6-2)**
12. 문법 검증 패스
```
