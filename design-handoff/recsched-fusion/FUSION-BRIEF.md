> **작성**: writer, 2026-07-24. 인풋: `CONTEXT-RECSCHED.md`(1/3) + `CONTEXT-RECRUITING-FULLVIEW.md`(2/3).
> **CD 에게**: §3 IA 제안과 §6 산출물 요청이 본론. §4–5 는 엔지니어링 배경(계약·정책 충돌)이며 디자인 판단에 영향을 주는 지점만 굵게 표시.

## 1. 목표 (사용자 의도)
1. 현재 두 서피스 — **리크루팅 위젯 풀뷰**(응답 분석)와 **`/admin/recruiting-scheduling`**(일정 조율) — 를 **하나로 융합**.
2. **프로필/어드민 경유 진입 제거**: `/admin/recruiting-scheduling` 및 계정 메뉴의 스케줄링 엔트리를 없애고, **리크루팅 위젯 풀뷰가 유일한 진입**.
3. 풀뷰가 현 스케줄링의 **list/calendar 뷰를 흡수·대체** — 이 과정에서 디자인 리프레시.
4. 참여자 표면(`/schedule/<token>`)은 **불변**(독립 공개 라우트).

## 3. 제안 IA — "리크루팅 여정 3단 퍼널" (CD 설계 대상)
리크루팅 풀뷰 내부를 현 2탭(요약/전체 데이터)에서 **여정 3단 최상위 탭**으로 재편:

```
[리크루팅 풀뷰 — sun 헤더, §F 셸 유지]
 ├─ ① 응답 (Responses)          ← 현 state 08 유지·리프레시
 │    조건 패널 · 분포 크로스탭 · 부합도 판단 · 응답 시트(요약/raw 서브탭 or 통합)
 │    행 선택 → [② 명단으로 보내기] 브리지 CTA        ★신규
 ├─ ② 명단 (Candidates)         ← 스케줄링 "리스트 뷰" 흡수
 │    후보 테이블(상태 pending/소통중/확정 · 그룹 · 원본 필드) · 벌크(확정/소통중/그룹배정)
 │    소스: ①에서 보낸 응답자 + 파일 업로드 + 시트 import(연동됨 카드)
 │    마스터링크 바 · 협업자 공유 → 풀뷰 헤더로 승격 검토
 └─ ③ 일정 (Schedule)           ← 스케줄링 "캘린더 뷰" 흡수
      캘린더(주/일, 컬러 블록, 가로접기) · 채팅(공지/발송×전체/그룹/개인, 멀티타일, 빨간콩)
      확정 로스터(읽기전용 그룹뷰) · 슬롯 에디터 모달
```

- **여정 서사**: 설문 발행 → 응답 들어옴(①) → 부합자 골라 명단에 올림(①→② 브리지) → 소통·확정(②) → 일정 잡고 안내(③) → 참여자는 마스터링크로. 지금 사용자가 머릿속으로 하던 흐름을 IA 가 그대로 표현.
- **위젯 카드**(캔버스 604×900)는 셋업 아코디언 그대로 — 발행 후 "전체 보기"가 이 퍼널의 관문.
- **진입 정리**: `/admin/recruiting-scheduling` → `/canvas?focus=recruiting`(풀뷰 자동 오픈) redirect 후 제거. 계정 메뉴 스케줄링 엔트리 제거. `/admin/recruiting-invitations`(슈퍼어드민 처리대)는 **운영 백오피스로 존치**(§5-D 결정에 따라 축소 가능).

## 4. 백엔드 브리지 (엔지니어링 방향 — 스펙화는 CD 산출물 확정 후)
1. **폼↔프로젝트 링크**: `sched_projects.form_id`(nullable, additive) — 리크루팅 폼당 sched 프로젝트 **1:1 자동 프로비저닝**(첫 브리지/첫 명단 진입 시 lazy 생성, title=폼 제목 동기). 기존 form 无 프로젝트(수동 생성분)는 form_id null 로 공존. G5 의 3중 개념 중 **폼이 anchor**, interview_projects pill 은 위젯 귀속용으로 별개 유지.
2. **브리지 API**(★핵심 신규): 선택 응답자 → sched_candidates 생성. **PII 는 서버에서만 흐름** — 서버가 Forms 응답에서 이름/전화/이메일을 뽑아 후보 행을 만들고, 클라에는 마스킹 정책에 따른 필드만 반환. 나머지 응답 컬럼은 `fields` jsonb 로 승계(추후 명단 테이블에서 원본 컨텍스트 유지). dedup = 기존 upsert 멀티키(email>phone>name) 재사용.
3. **초대 플로우 재배선**(§5-D 결정 종속): (A)안이면 invitations PATCH `sent` 시 서버가 자동 인제스트 — 처리대가 곧 브리지 게이트. (B)안이면 invitations 를 건너뛰고 유저 셀프 브리지.
4. **접근 통일**: 융합 서피스의 게이트 = **폼 소유자 OR 그 org 멤버** (스케줄링의 `getSchedulingAccess` 를 form-anchor 로 확장). 겸사겸사 `sched_*` 에 **org_id 도입**(B4 부채 정리) 권장 — 코드 owner-chain 을 org_id 컬럼+인덱스로 단순화.
5. **보존 계약(불변)**: 참여자 라우트/폰게이트/HMAC 쿠키 · 채팅 fan-out payload(is_announcement×batch_id×private) · 슬롯 fan-out 모델 · 소스 인제스트 upsert · Realtime/폴링 · PII 블랭킹 파이프(responses) · admin-proxy 토큰 라우팅.

## 5. 🔴 결정 필요 (사용자+CD — 스펙 확정 전)
- **(D1) PII·브리지 모델**: 응답 연락처는 유저 비공개(어드민 대행)가 현 정책. 그런데 명단(②)은 연락처가 보이는 표면.
  - **A. 어드민 게이트 유지(권장·보수)**: 브리지 = 초대요청 → 슈퍼어드민 승인(sent) 시 **자동 인제스트**. 유저는 ②에서 이름·상태만, 연락처 컬럼은 마스킹(슈퍼어드민만 평문). 수작업 제거되면서 정책 불변.
  - **B. 셀프서브(정책 완화)**: 유저가 직접 브리지, ② 연락처 평문. 마찰 최소지만 "연락처는 대행" 정책 폐기 = 제품 방침 변경.
  - ※ 현 스케줄링(파일/시트 직접 업로드 소스)은 유저 본인 데이터라 연락처 노출이 자연스러움 — **소스별 이원 정책**(업로드분=평문, 응답 브리지분=마스킹)이 A 안의 현실적 형태.
- **(D2) 풀뷰 프레임 치수**: 현 셸 max **1600×900** vs 캘린더 카드 **1020px 높이** — ③이 현 프레임에 안 들어감. CD 가 결정: 프레임 키우기(90vh 활용) vs ③ 내부 스크롤/밀도 조정 vs 캘린더·채팅 분리 레이아웃.
- **(D3) 협업자 공유·마스터링크의 자리**: 탭 안(②) vs 풀뷰 헤더 승격(전 탭 공통) — 헤더 승격 권장(공유는 여정 전체의 관심사).
- **(D4) `/admin/recruiting-invitations` 처리대**: D1-A 면 존치(승인=브리지 게이트), D1-B 면 대폭 축소/제거.
- **(D5) 폼 다중 프로젝트**: 폼 1:1 프로젝트로 충분한가, 폼 하나에 회차(프로젝트) 여러 개가 필요한가.

## 6. CD 산출물 요청 (CD-DELIVERABLE-RULES 준수)
- **프레임 제안 목록** (기존 recsched 7프레임 + fullview state 08 을 흡수·대체):
  - N1 풀뷰-응답 탭(state 08 리프레시 + ①→② 브리지 CTA/선택 모드)
  - N2 풀뷰-명단 탭(리스트 뷰 흡수: 테이블·벌크바·상태칩·그룹뷰·소스 인테이크·연동됨 카드·마스터링크)
  - N3 풀뷰-일정 탭(캘린더+채팅+로스터 — **D2 치수 결정 포함**)
  - N4 브리지 플로우(응답 선택→명단 보내기 모달/확인 — D1 모델 반영)
  - N5 상태 커버: 각 탭 empty/loading/error + Google 재인증/미연동 + (기존 미작화) 슬롯 생성 모드·OAuth 바운스
  - 헤더: 3탭 내비 + 공유·마스터링크 승격안(D3) + 죽어있던 CSV/새로고침/프로젝트 pill 재배치
- **비주얼**: sun 톤 기준 통일(응답=sun 인데 현 스케줄링 어드민도 sun — 충돌 없음, 참여자 sky 유지). 기존 사용자-승인 이탈(멀티타일 채팅·가로접기·컴팩트 위계·빨간콩 등)은 **동작 요구사항으로 승계** — 형태는 CD 재량.
- 산출 형식: 기존 규칙대로 README+HANDOFF+BUILD-SPEC+`.dc.html`(support.js 렌더) 번들.
