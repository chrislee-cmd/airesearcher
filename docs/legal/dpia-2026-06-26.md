# Data Protection Impact Assessment (DPIA)

- **Controller**: Meteor Research Inc.
- **버전**: 1.0 (2026-06-26 — PR-SEC12 draft, awaiting legal review)
- **법적 근거**: GDPR Article 35 (DPIA) + Art. 36 (DPA prior consultation)
- **범위**: 본 DPIA 는 AI Researcher 의 **고위험 처리 활동** 에 한정.
  저위험 처리 (가입, 결제, OAuth 토큰 보관) 는 ROPA 만 적용
- **상위 문서**:
  - `docs/legal/ropa-2026-06-26.md` (전체 처리 활동 매핑)
  - `docs/security-audit-data-flow.md` (mermaid 데이터 흐름)
  - `docs/security-audit-risk-matrix.md` (발견 ID + 잔여 위험)

> **상태**: 워커 초안. DPO consultant + 사내 legal sign-off 전 internal-only. EU launch 게이트 D+10주 — 그 전 review 완료 필수.

## 1. DPIA 필수 여부 판단 (Art. 35(3) + WP248)

다음 trigger 중 하나라도 해당 시 DPIA 필수. AI Researcher 의 처리 활동은 **3개** 가 해당:

| Trigger | 해당 여부 | 근거 |
|---|---|---|
| (a) 자동화된 평가 / 프로파일링 + 법적 효과 | **부분** | LLM 이 인터뷰 분석 결과를 도출 — 법적 효과는 없으나 사용자 의사결정에 영향 |
| (b) 대규모 special category 처리 | **잠재** | 인터뷰 내용에 보건/심리/소수자 발화 포함 가능 (Art. 9) |
| (c) 공개적으로 접근 가능 영역의 systematic monitoring | 해당 없음 | 우리 서비스는 자발적 업로드 |
| (d) 신기술 사용 | **해당** | 생성형 AI (OpenAI / Anthropic / Deepgram / ElevenLabs / Twelvelabs) — Art. 35 권고 명시 |
| (e) Vulnerable 데이터 주체 | **잠재** | 인터뷰 respondent 중 minor / patient / employee 포함 가능 (사용자가 통제) |

→ **결론: DPIA 필수.** 이하 §2~§7 진행.

## 2. 처리 활동 description (Art. 35(7)(a))

### 2.1 고위험 처리 #1 — LLM 기반 인터뷰 분석 (insights / chat / reports)

- **입력**: 사용자가 업로드한 인터뷰 transcript (text), 음성/영상 raw (변환 후 text), 사용자가 지정한 리서치 질문/가이드
- **처리**: Anthropic Claude / OpenAI GPT 에 prompt 송신 → quote 추출, 클러스터링, contradiction 탐지, RQ 별 종합 → DB 저장
- **출력**: `insights_quotes.text` + `insights_clusters.label` + `interview_chunks` + chat 응답 + 리포트 markdown
- **사용자 작업**: UI 에서 결과 조회 + 편집 + workspace artifact 로 등록
- **데이터 카테고리**: Sensitive PII (음성/영상 raw → text), 3자 발화 포함 가능, **잠재적 Art. 9 special category** (보건/심리/정치/종교/소수자 발화 등 인터뷰 주제에 따라)
- **데이터 주체**:
  - 사용자 (org member) — 직접 동의
  - 3자 (respondent) — **사용자가 통지 의무 보유 (Art. 14)**. 공동 controller 가능성 (Art. 26) 검토 필요
- **법적 근거**:
  - 사용자: Art. 6(1)(b) 계약 이행 + Art. 6(1)(a) `llm_processing` 명시 동의
  - 3자: 사용자 가 controller 로서 Art. 6 / Art. 14 충족 책임 (현재 `/use-policy §4` 에 명시)
  - Special category 발견 시: Art. 9(2)(a) 명시 동의 — 사용자가 respondent 로부터 받아야 함
- **신기술**: 생성형 AI — 학습 데이터 미사용 (OpenAI API tier default + Anthropic zero-retention default)

### 2.2 고위험 처리 #2 — 음성 / 영상 처리 (transcripts / video / voice)

- **입력**: 음성 파일 (`audio-uploads` Storage), 영상 파일 (`video-uploads` Storage), 실시간 audio stream (LiveKit)
- **처리**: Deepgram / ElevenLabs (전사), OpenAI Realtime (실시간 전사 + 번역), Twelvelabs (영상 indexing)
- **데이터 카테고리**: Sensitive PII + **생체정보 가능성 (Art. 9(1))** — 음성 자체가 식별 가능 생체정보로 해석될 여지가 있음 (CJEU 판례 분기)
- **데이터 주체**: 발화 주체 (사용자 본인 + 3자)
- **신기술**: 음성/영상 ML pipeline (제 3 처리자 의존)

### 2.3 고위험 처리 #3 — 데스크 리서치 자동화 (desk)

- **입력**: 사용자 지정 키워드 + 외부 article 크롤 결과
- **처리**: Anthropic Claude 가 RQ decompose → claim 추출 → 종합 markdown
- **데이터 카테고리**: 사용자 키워드 (저위험) + crawled article (3자 source — public 가정)
- **위험**: prompt injection 표면 (사용자 input 이 system prompt 분리 없이 LLM 에 도달 — PR-SEC9 에서 fix)

## 3. 자동화된 결정 (Art. 22) 평가

| 활동 | 자동화된 결정 / 프로파일링? | 사용자 법적·중대 영향? | 인간 개입 여부 |
|---|---|---|---|
| LLM insights 분석 결과 | **분석 (decision support)** — 자동 결정 아님 | 없음 (결정은 사용자가 함) | 사용자가 결과 검토/편집 |
| 음성 전사 | 결정 아님 | 없음 | n/a |
| 결제 처리 | 결정 (구매) — 사용자 액션 | 약정 효과 (결제) | 사용자 명시 클릭 |
| 무료 trial 자격 (fingerprint) | **자동 거절 가능** | 무료 credit 거부 | 인간 검토 없음 → **부분 Art. 22 해당** |

**Trial fingerprint 거절** 의 경우 Art. 22(1) "자동화된 처리에 기반한 결정으로 법적 효과 또는 유사한 중대 영향이 있는 결정에 따르지 않을 권리" 적용 가능성 평가:
- "중대 영향" 해당 — 무료 credit 거부는 보통 minor (유료 가입 가능). but 향후 더 strict 한 anti-abuse 가 추가되면 평가 변경
- **완화**: 사용자가 수동 review 요청 가능한 channel 제공 (privacy@... 메일) + 향후 unblock UI 추가 평가

기타 활동은 모두 인간 개입 가능 / 결정 아님 → Art. 22 비대상.

## 4. 필요성 + 비례성 평가 (Art. 35(7)(b))

| 처리 | 필요한가? | 대안? | 비례? |
|---|---|---|---|
| LLM analysis | **Yes** — 핵심 가치 제안 | 사용자 수동 분석 (비효율). 부분 대안: 사용자 in-tenant LLM 호출 | Yes — 사용자가 cell 단위로 trigger, 결과 편집/삭제 가능 |
| 음성 raw → 처리자 송신 | **Yes** — 전사가 핵심 기능 | local Whisper.cpp 실험 (latency / 정확도 trade-off 미달) | Yes — private bucket + signed URL + processor DPA |
| 3자 (respondent) 발화 처리 | **Yes** — 인터뷰 분석의 본질 | n/a (대안 없음) | **Partial** — Art. 14 통지 의무 사용자 측 + 공동 controller 책임 가능 |
| Mixpanel email = distinct_id | No — 행동 분석 자체는 필요하나 PII 결합은 선택 | UUID-only distinct_id (현재 검토) | Improvable — 권고 §6 |
| trial_fingerprints IP 보존 | Partial — 7일이면 충분 | hash-only (이미 적용), 더 짧은 retention | Yes (7일 cron) |

**결론**: 핵심 처리는 비례적. 대안 가능한 항목 (Mixpanel distinct_id, audit_log 365일, trial 7일) 은 권고에 반영.

## 6. 완화 조치 (Art. 35(7)(d)) — 본 PR (SEC12) 의 기여분

PR-SEC12 가 본 DPIA 에 기여하는 조치:

1. **Sentry beforeSend PII redactor** (src/lib/sentry-pii.ts):
   - email / JWT / Bearer / Korean phone 패턴 redact
   - PII 라우트 (`/api/account/`, `/api/transcripts/`, …) body 전체 drop
   - cookies / Authorization 헤더 redact
   - 결과: 에러 추적이 PII 누출 없이 동작
2. **audit_log event_type 확장** (src/lib/audit.ts):
   - `data_export_requested` + `data_export_completed` → Art. 15 / 20 행사 추적
   - `account_deletion_requested` + `account_deletion_completed` → Art. 17 행사 추적
   - `consent_version_updated` → Art. 7(1) 동의 demonstrability
   - `rls_violation_attempted` / `admin_impersonation` / `config_changed` → 운영 forensic
3. **ROPA / DPIA 문서화** (이 디렉토리):
   - Art. 30 ROPA + Art. 35 DPIA 형식 충족
   - CISO / DPO consultant review input

## 7. 권고 (필수 / 권장)

### 7.1 필수 (EU launch 게이트 — D+10주)

1. **PR-SEC9 / SEC10 / SEC11 / SEC15 머지**: prompt injection, LLM ZDR, Supabase region, 3자 통지 페이지
2. **DPA 서면 체결 확인**: Supabase / OpenAI / Anthropic / Deepgram / ElevenLabs / LiveKit / Twelvelabs / Stripe / LS / Mixpanel / Google / Notion — DPA URL 과 서면본 모두 별 트래커 (DEBT 또는 legal 시스템)
3. **DPO consultant 1회 review**: 본 DPIA + ROPA + privacy policy v2 일괄 검토
4. **침해 통지 절차** (`docs/security-incident-response.md`): 72시간 GDPR Art. 33 절차 + KR PIPA 24시간 절차 통합

### 7.2 권장 (D+12~16주)

1. **Mixpanel distinct_id 를 UUID-only 로 전환** — email 결합 해제
2. **음성 raw retention 정책** — 처리 후 N일 hard delete (현재 사용자 삭제까지)
3. **MFA / WebAuthn 도입** — Supabase 지원, UI 미구현
4. **외부 침투 테스트** — Phase 0 완료 후 의뢰
5. **`config_changed` audit 자동 wiring** — Vercel/Supabase 운영 액션을 GitHub Actions / Vercel webhook 로 audit_log 에 흘리기

### 7.3 검토 후 결정 필요

- **공동 controller (Art. 26) vs processor**: respondent 발화 처리에서 Meteor 의 역할. 현재 use-policy 는 사용자 controller / Meteor processor 모델. 법무 의견 필요
- **음성 = biometric Art. 9 여부**: CJEU 판례 추적, conservative 접근으로는 special category 가정 → `llm_processing` consent 강화

## 9. DPA prior consultation (Art. 36)

§5 의 잔여 위험 평가 결과 **high residual risk 가 모두 후속 PR 로 완화 예정** → Art. 36 prior consultation (사전 협의) 의무 미발동. 단:

- PR-SEC9 (prompt injection) / PR-SEC11 (Supabase region) 머지 전 launch 강행 시 → Art. 36 의무 발동 가능
- EU launch 게이트 (모든 P0 + P1 머지 + DPO sign-off) 가 자동 차단 역할
